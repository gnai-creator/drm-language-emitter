# 004 - Source Code Audit

Report date: 2026-07-11

Scope: source audit of `src/drm_language_emitter`, related tests, and directly relevant scripts. This report is based on static review plus local test execution. Existing uncommitted changes were present before this report in `src/drm_language_emitter/model.py` and `docs/report/003_report_2926_07_11.md`; they were treated as user work and not modified.

## Verification Commands

```powershell
pytest
$env:PYTHONPATH='.;src'; pytest
```

Results:

- Current result after fixes: plain `pytest` passes locally.
- Historical result at audit time: `pytest` failed during collection with `ModuleNotFoundError: No module named 'world_model'`; with manual `PYTHONPATH='.;src'`, 3 core model tests failed because `DRMStateInitializer` had no `forward`.

## Executive Summary

The repository is organized clearly, with a compact DRM core, explicit diagnostics, transformer/world-model baselines, and smoke tests for the main workflows. The two blocking issues found during the audit have been fixed: the state initializer is callable again, and plain `pytest` can import the top-level `world_model` package.

## Findings

### Critical: `DRMStateInitializer` cannot be called - fixed

Evidence:

- `src/drm_language_emitter/model.py:31` defines `DRMStateInitializer._forward_impl`.
- `src/drm_language_emitter/model.py:56` calls `self.initializer(batch, input_ids.device)`.
- `src/drm_language_emitter/generation.py:22` calls `model.initializer(batch, input_ids.device)`.
- There is no `forward` method on `DRMStateInitializer`.

Status:

Fixed. `DRMStateInitializer.forward` exists and the core forward/generation tests pass.

### High: Default pytest cannot import top-level packages - fixed

Evidence:

- `pyproject.toml:24-27` configures `pythonpath = ["src"]`.
- `tests/test_tiny_world_dataset_smoke.py:6` imports `world_model.tiny_world`.
- `world_model/` is a top-level package outside `src`.

Status:

Fixed. `pyproject.toml` includes both `src` and `.` for pytest and package discovery includes `world_model*` and `transformer*`.

### Medium: `metric_rank=0` still pays an impossible U-floor penalty - fixed

Evidence:

- `src/drm_language_emitter/config.py:75` allows `metric_rank` to be zero.
- `src/drm_language_emitter/metric.py:30-31` returns a zero-width `metric_u` for rank zero.
- `src/drm_language_emitter/model.py:97` records zero U norms when `metric_u.numel()` is false.
- `src/drm_language_emitter/model.py:123-128` computes `metric_u_floor_loss` from those zero norms.
- `src/drm_language_emitter/model.py:162-164` adds the floor loss whenever `lambda_metric_u_floor` is nonzero.
- The default `lambda_metric_u_floor` is `0.001` at `src/drm_language_emitter/config.py:51`.

Status:

Fixed. The U-floor diagnostic becomes zero when `metric_rank == 0`, and the U-floor loss is not added to `aux_losses` in that case.

### Medium: Configuration exposes unsupported or no-op options

Evidence:

- `src/drm_language_emitter/config.py:20` defines `use_toroidal_state`.
- `src/drm_language_emitter/dynamics.py:52-53` defines `toroidal_pairs`, but `StateUpdater.forward` at `src/drm_language_emitter/dynamics.py:42-49` does not branch on `use_toroidal_state`.
- `src/drm_language_emitter/config.py:37` defines `tie_embeddings`.
- `src/drm_language_emitter/emitter.py:10-16` and `src/drm_language_emitter/emitter.py:29-55` create independent embedding and output layers; no implementation references `tie_embeddings`.

Impact:

Experiment configs can claim toroidal state or tied embeddings without changing runtime behavior. This is risky for research reporting because metrics may be attributed to features that were never active.

Recommended fix:

Either implement these options or remove/deprecate them. Add tests that assert the expected behavior changes when each flag is enabled.

### Low: `DRMConfig.from_dict` silently ignores unknown keys - fixed

Evidence:

- `src/drm_language_emitter/config.py:156-160` filters input config data to dataclass fields and drops the rest.

Status:

Fixed. `DRMConfig.from_dict` raises `ValueError` for unknown fields.

### Low: `torch.load` has no payload validation boundary - fixed

Evidence:

- `src/drm_language_emitter/checkpoint.py:11-17` loads any checkpoint and assumes `payload["config"]` and `payload["model"]` exist.

Status:

Fixed for structure validation. Checkpoints should still be treated as trusted unless a fully hardened loading path is added.

## Positive Notes

- The core model is small and modular: embedding, direction field, metric, flow, updater, risk, and emitter are separate modules.
- Diagnostics are comprehensive and JSON conversion is centralized in `src/drm_language_emitter/utils.py`.
- Tests cover no-attention guarantees, loss helpers, metric behavior, generation shape, geometry diagnostics, and world-model dataset smoke checks.
- The README clearly states the research status and avoids overclaiming production readiness.

## Recommended Fix Order

1. Decide whether `use_toroidal_state` and `tie_embeddings` are supported features or stale config fields.
2. Add explicit tests for any future implementation of those flags.
3. Continue treating checkpoints from untrusted sources as unsafe unless a hardened loader is introduced.

## Current Risk Rating

Overall: medium.

The current working tree passes the local test suite. Remaining risk is mostly stale experimental flags and longer-run validation, not immediate test breakage.
